Designing a shared language for prompts and filters.

Recruiters already move between natural language and structured filters when they source. This concept makes both inputs produce the same editable search state: visible criteria, explicit rules, and candidate cards that explain why someone matched.

One-page brief

The thinking before the screens

Problem framing

Today's search splits across two surfaces: a freeform prompt and a structured filter panel. They both feed the same results, but the recruiter can't see which criteria came from where, which ones are firm rules versus loose preferences, or what the system inferred along the way. Editing usually means starting over. The cost isn't typing speed, it's the gap between what the recruiter meant and what the result set actually represents.

A recruiter should be able to see which criteria shaped a result, and change those criteria without starting over.

Key assumptions

  • The prompt parser is useful but imperfect. Extractions need to be visible and removable per item, not in bulk.
  • Sourcing is iterative. The work is usually converging from a thousand candidates to thirty, not getting the right answer on the first try.
  • Result count is the recruiter's main feedback signal. It belongs on screen at all times.
  • Geography is the hardest filter to design. Disambiguation and progressive disclosure beat a wall of dropdowns.
  • Recruiters accept inferred suggestions if those suggestions are clearly labeled and easy to remove.

Design principles

  • Prompt output starts as editable draft criteria. Drafts wait for Enter. Nothing applies on a timer.
  • Prompt and filter inputs land in the same applied-state row. The chip row is the editor for both.
  • Every chip shows its origin. Recruiters can tell what they typed, what they clicked, and what the system inferred.
  • Conflicts ask for a choice instead of blocking the search. Contradictions appear as inline questions with the result count for each option.
  • Rule vs preference is part of every chip. Solid borders are rules (candidates must pass). Dashed borders are preferences (they shape ranking). Inferred chips start as preferences until confirmed. One click promotes or demotes any chip.

What "good" looks like

  • A recruiter goes from an open role to roughly thirty plausible candidates in under a minute, with a clear sense of why each one appears.
  • Editing a result set never requires going back to the prompt. Adjustments happen on the chip row.
  • The system doesn't ask for confirmation. It asks the recruiter to choose between concrete options, with the count for each option visible.
  • Zero-result and overly-broad states behave like guided refinement, not error messages. The next suggested step is always one small change away.

What I cut, and why

No algorithmic explanations of why specific candidates were ranked a certain way. For an MVP, showing which chips matched is more honest than inventing a confidence score. No separate "advanced filter" surface: if a filter doesn't deserve a chip, it doesn't need to exist yet. No undo stack: removing chips one by one already serves as edit history. No saved-search management screen in this round, since recent searches on the start screen cover most of the use case.

Notes
  • Parsing is visible The recruiter sees each chip form as the prompt is typed, instead of waiting for a single result list to appear.
  • Two signals per draft chip Color says where the chip came from (prompt, filter, inferred). Border style says how it will behave once applied (solid = rule, dashed = preference). Both are readable before the recruiter commits.
  • Predicted behavior is editable now The parser proposes "Senior" and "AI/ML industry" as preferences because they read as soft signals. A tap flips any chip between rule and preference before pressing Enter, so the recruiter doesn't need to wait until after the search runs.
  • Nothing applies until Enter No pause-to-commit, no smart timeout. Inferred chips need a second tap to confirm before they affect results.
  • Failure mode If the parser returns zero chips, the search falls back to keyword matching and tells the recruiter that's what happened.

Coda

What I'd build first, and what's next

MVP scope

  • → Chip parser for: title, skills, location, seniority, work-pref, industry
  • → Source-tagged chip row ([p]/[f]/[i]) + inline removal
  • → Rule vs preference per chip (solid/dashed border) with right-click promote/demote
  • → Hierarchical geography (country/region/city + radius + multi-location)
  • → Search history ladder showing the path from broad to focused, with one-click rewind
  • → Loosen/Tighten menus with predicted count impact per option
  • → Three-line "why this candidate": passed rules · matched preferences · gaps
  • → Work-pref/seniority conflict resolution with count-per-choice

Round two

  • → Boolean grouping for advanced power users (still in chip language)
  • → Candidate-side "why was I shown" parity for trust
  • → Saved-search collaboration (share applied state with hiring manager)
  • → Per-chip weight tuning for power users (default: equal weighting within behavior tier)
  • → Conflict templates (common patterns like remote-vs-on-site stored as 1-click resolutions)
The main design decision is to make prompt output and filter selections editable in the same place. Once active criteria are visible, conflict handling, result explanations, and search refinement can all reuse the same chip model.
